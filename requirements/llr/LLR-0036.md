## Description

The system shall detect SQLite lock contention (`SQLITE_BUSY`, `SQLITE_LOCKED`) on any
write operation, report a clear actionable error to stderr, and exit non-zero. Silent
failure on lock contention is explicitly prohibited.

## Rationale

The MCP server holds a persistent WAL reader connection to `cache.db`. Commands such as
`req init`, `req scan`, and `req remove` open a write connection to the same file. When
the file is already locked, `rusqlite` returns an error that the current code discards,
causing the command to exit 0 with no output. This silent failure is indistinguishable
from success and masks data integrity issues. An explicit, user-readable error with a
suggested remedy eliminates the confusion.

## Acceptance Criteria

- [ ] Any write path that receives `SQLITE_BUSY` or `SQLITE_LOCKED` from rusqlite shall
      print a message to stderr in the form:
      `error: cache.db is locked by another process (likely the MCP server). Stop the
      MCP server or use --wait <seconds> to retry.`
- [ ] The affected command exits with a non-zero status code in all lock-contention cases
- [ ] `--wait <seconds>` option (available on all write commands: `init`, `scan`,
      `remove`, `import`, `import-ai`, `import-reqif`) causes the system to retry the
      connection at 100 ms intervals until either the lock is released or the timeout
      expires; on timeout the same error message is printed and the command exits non-zero
- [ ] Default value of `--wait` is `0` (fail immediately; no silent retry)
- [ ] The lock-detection logic is implemented in `req_engine` so it is available to both
      the CLI and MCP write paths
- [ ] No code path may swallow or ignore `SQLITE_BUSY` / `SQLITE_LOCKED` errors; this is
      enforced by a clippy lint or equivalent static check where possible
