## Description

Requirements shall support arbitrary key-value metadata without requiring code changes, enabling users to add custom fields for their specific workflows.

## Rationale

Different organizations have different metadata needs (ASIL levels, safety classes, verification methods, etc.). Hardcoding these fields would make the tool inflexible. A key-value approach allows customization without forking.

## Acceptance Criteria

- [ ] `attributes` field on Requirement struct (HashMap<String, String>)
- [ ] Attributes serialized to Markdown frontmatter
- [ ] Attributes preserved on import/export roundtrip
- [ ] No validation on attribute keys (user freedom)
- [ ] CLI `--attr key=value` option for `req new`
- [ ] CLI `--attr key=value` option for `req import`

## Implementation

### Markdown Format

```yaml
---
id: LLR-0015
type: llr
status: approved
parent: HLR-0010
attributes:
  asil: "ASIL-B"
  verification_method: "test"
  reviewer: "john.doe"
  custom_field: "any value"
---
```

### JSON Format

```json
{
  "id": "LLR-0015",
  "attributes": {
    "asil": "ASIL-B",
    "verification_method": "test"
  }
}
```

### CLI Usage

```bash
req new llr "Safety requirement" --parent HLR-0001 --attr asil=ASIL-B --attr reviewer=john.doe
