## Description

The scanner shall recognise a new tag `CRITERION: LLR-XXXX #N` in source files, linking
a test function body to the Nth `- [ ] ...` item in the named LLR's acceptance criteria
block (1-based index). The engine shall expose a method that reports which criteria items
are linked to at least one test and which are unlinked.

## Tag Syntax

```rust
// CRITERION: LLR-0042 #3
#[test]
fn test_boundary_case() { ... }
```

`#3` refers to the third `- [ ] ...` bullet in `LLR-0042.md`, counting top-to-bottom,
starting at 1. The tag may appear on any line; the scanner associates it with the
enclosing function via proximity (within 5 lines before the `fn` keyword).

## Implementation Notes

- New `TagType::Criterion` variant in `req_engine/src/scanner.rs`
- New field `criterion_index: Option<usize>` on `FoundTag` (defaults to `None` for all
  existing tag types)
- New regex: `r"CRITERION:\s*([A-Z]+-\d+)\s*#(\d+)"`
- New cache table (schema version 2):
  ```sql
  CREATE TABLE IF NOT EXISTS criterion_links (
      req_id          TEXT    NOT NULL,
      criterion_index INTEGER NOT NULL,
      file            TEXT    NOT NULL,
      line            INTEGER NOT NULL,
      PRIMARY KEY (req_id, criterion_index, file, line)
  );
  ```
- Schema migration: if `schema_version < 2` in the existing database, add the new table
- Engine method: `pub fn audit_criteria(&self) -> Result<Vec<CriteriaReport>>`
- Criterion count in an LLR file: count lines matching `^\s*-\s+\[[ xX]\]`

## Acceptance Criteria

- [ ] Scanner recognises `CRITERION: LLR-XXXX #N` and stores in `criterion_links` table
- [ ] `audit_criteria()` returns one `CriteriaReport` per LLR that has acceptance criteria
- [ ] Each `CriteriaReport` lists every criterion with `linked: bool` and test source location
- [ ] LLRs where the max `#N` referenced exceeds the `- [ ]` count produce a Warning
- [ ] Existing tag types (`REQ:`, `VERIFIES:`) are unaffected
- [ ] Cache schema migration from version 1 to 2 completes without data loss
