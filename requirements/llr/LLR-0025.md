## Description

The `req_mcp` binary implements an MCP server using `rmcp` v1.2.0 with stdio transport. It registers all `req` engine operations as MCP tools and resources using the `#[tool]` and `#[tool_router]` macros, enabling AI models to invoke them via JSON-RPC over stdin/stdout.

## Implementation

- Location: `req_mcp/src/`
- Server handler: `req_mcp/src/server.rs` — implements `ServerHandler` trait via `#[tool_router]`
- Tool handlers: `req_mcp/src/tools.rs` — one method per MCP tool, annotated with `#[tool]`
- Resource handlers: `req_mcp/src/resources.rs` — MCP resource URIs
- Binary: `req_mcp/src/main.rs` — initialises `ReqEngine` and starts `rmcp` stdio server
- Transport: `rmcp::transport::stdio` (default)
- Each tool method accepts a JSON-deserializable input struct and returns a JSON-serializable output struct; JSON Schema is auto-generated by `rmcp-macros`

## MCP Tools

### Requirements lifecycle

| Tool name | Engine method | Input | Output |
|-----------|--------------|-------|--------|
| `req_init` | `ReqEngine::init()` | `{}` | status message (no-op if already initialised) |
| `req_create_requirement` | `ReqEngine::create_requirement()` | `{ "type": "hlr\|llr\|tst", "title": "...", "parent": "...", "status": "draft\|approved\|..." }` | `Requirement` |
| `req_remove` | `ReqEngine::remove_requirement()` | `{ "id": "LLR-XXXX", "cache_only": bool, "force": bool }` | `RemoveResult` |
| `req_import` | `ReqEngine::import()` | `{ "input": "/path", "format": "markdown\|json", "provenance": "..." }` | `Vec<Requirement>` |
| `req_import_ai` | `ReqEngine::import_ai()` | `{ "input": "/path", "dry_run": bool, "provenance": "..." }` | `ImportResult` |
| `req_migrate` | `ReqEngine::migrate()` | `{ "dry_run": bool }` | `MigrateResult` |

### Query & analysis

| Tool name | Engine method | Input | Output |
|-----------|--------------|-------|--------|
| `req.scan` | `ReqEngine::scan()` | `{ "clear": bool }` | `ScanResult` |
| `req.coverage` | `ReqEngine::coverage()` | `{}` | `Coverage` (traceability: HLR→LLR linkage, LLR implementation %) |
| `req.gaps` | `ReqEngine::gaps_full()` | `{}` | gap list (structural + import staleness) |
| `req.check` | `ReqEngine::validate()` | `{}` | issue list |
| `req.trace` | `ReqEngine::trace(id)` | `{ "id": "LLR-XXXX" }` | trace tree |
| `req.impact` | `ReqEngine::impact(id)` | `{ "id": "HLR-XXXX" }` | impact list |
| `req.list` | `ReqEngine::list()` | `{ "type": "hlr\|llr\|tst", "status": "..." }` | requirement list |
| `req.export` | `ReqEngine::export()` | `{ "format": "json\|ai-context\|markdown", "id": "..." }` | export payload |

### AI output integrity audits

| Tool name | Engine method | Input | Output |
|-----------|--------------|-------|--------|
| `req_audit_triviality` | `ReqEngine::audit_triviality()` | `{ "id": "LLR-XXXX" }` (optional) | `Vec<TrivialityReport>` |
| `req_audit_criteria` | `ReqEngine::audit_criteria()` | `{ "id": "LLR-XXXX" }` | `CriteriaReport` |
| `req_audit_mutation` | `ReqEngine::audit_mutation(report)` | `{ "report": "/path" }` | `MutationReport` |
| `req_audit_coverage` | `ReqEngine::audit_coverage(report)` | `{ "report": "/path" }` | `Vec<LineCoverageScore>` |
| `req_audit_export_context` | `ReqEngine::audit_export()` | `{ "id": "LLR-XXXX", "mutation": "/path?", "coverage": "/path?" }` | `AuditBundle` |
| `req_audit_independence` | `ReqEngine::audit_independence()` | `{ "id": "LLR-XXXX" }` (optional) | `IndependenceResult` |
| `req_check_provenance` | `provenance::check_all_provenance()` | `{}` | `Vec<ProvenanceViolation>` |

The `req_audit_coverage` tool parses a `cargo llvm-cov --json` report and returns per-LLR line-hit percentages. The `report` field is the path to the JSON file produced by `cargo llvm-cov --json --output-path <file>`. Returns an error if the file is absent or malformed.

## MCP Resources

In addition to the tools listed above, the following resources are exposed:

| URI | Engine call | Description |
| --- | --- | --- |
| `req://coverage` | `ReqEngine::coverage()` | Traceability coverage (same as `req.coverage` tool) |
| `req://gaps` | `ReqEngine::gaps()` | Traceability gaps |
| `req://requirements/{type}` | `ReqEngine::list_requirements(type)` | All requirements of a given type |
| `req://requirements/{type}/{id}` | list + filter by id | Single requirement |

## Acceptance Criteria

- [ ] `req-mcp-server` binary starts and responds to MCP `initialize` message
- [ ] All 21 MCP tools are registered and return correct JSON output when called
- [ ] All 4 MCP resources (`req://requirements/...`, `req://coverage`, `req://gaps`) are accessible
- [ ] `req_audit_coverage` returns `LineCoverageScore` array for a valid `cargo llvm-cov --json` report
- [ ] `req_audit_coverage` returns a tool error (not a server crash) when the report file is missing
- [ ] `req_create_requirement` creates a requirement file on disk and returns it as JSON
- [ ] `req_remove` purges cache entries and reports dependents when `force` is false
- [ ] `req_import` imports requirements from Markdown and JSON files
- [ ] `req_import_ai` forces all imported requirements to `draft` status
- [ ] `req_migrate` reports migrated files and supports dry-run mode
- [ ] `req_audit_triviality` returns an array of `TrivialityReport`
- [ ] `req_audit_criteria` returns a `CriteriaReport` for a given LLR
- [ ] `req_audit_mutation` returns a `MutationReport` for a valid cargo-mutants report
- [ ] `req_audit_export_context` returns a full `AuditBundle` for a given LLR
- [ ] `req_audit_independence` returns an `IndependenceResult`
- [ ] `req_check_provenance` returns an array of `ProvenanceViolation`
- [ ] `req_init` reports already-initialised when the project exists
- [ ] `req_export` supports `json`, `ai-context`, and `markdown` formats with optional `id`
- [ ] `req_gaps` includes import staleness flags
- [ ] Server exits cleanly on stdin close
- [ ] `cargo test -p req_mcp` passes (including in-process transport tests)
