## Description

Test suite verifying the MCP server implementation in `req_mcp`. Tests use the `rmcp` in-process transport to invoke tools and resources without spawning a child process.

## Test Cases

### TC-001: Server Initialisation

Verify the MCP server starts correctly:

- Responds to the MCP `initialize` message
- Returns correct `serverInfo` (name: `req-mcp-server`)
- Lists all 8 registered tools in `tools/list` response

### TC-002: req.scan Tool

Verify the `req.scan` MCP tool:

- Accepts `{ "dirs": ["src"] }` as input
- Returns a `ScanResult` JSON object
- Fails gracefully if the project is not initialised

### TC-003: req.coverage Tool

Verify the `req.coverage` MCP tool:

- Accepts `{}` as input
- Returns a `Coverage` JSON object with percentage fields
- Fields match `Coverage` struct schema

### TC-004: req.gaps Tool

Verify the `req.gaps` MCP tool:

- Returns a JSON array of gap descriptors
- Each gap has `type` and `id` fields

### TC-005: req.list Tool

Verify the `req.list` MCP tool:

- Accepts `{ "type": "hlr" }` filter
- Returns only requirements of the requested type
- Accepts `{ "status": "draft" }` filter

### TC-006: Resource Access

Verify MCP resource URIs:

- `req://coverage` returns current coverage JSON
- `req://requirements/hlr` returns all HLR entries
- `req://requirements/hlr/HLR-0001` returns a single requirement or 404
