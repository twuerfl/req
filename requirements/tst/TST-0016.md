## Description

Tests for the `CRITERION:` tag scanning, cache storage, and `ReqEngine::audit_criteria`
facade method introduced by LLR-0029.

## Test Cases

### TC-001 — Scanner parses CRITERION tag and stores in cache
Set up a `TempDir` with `ReqEngine::init`. Write a source file containing:
```rust
// CRITERION: LLR-0001 #2
#[test]
fn test_criterion() {}
```
Run `engine.scan(None, false)`. Query the `criterion_links` table directly.
Assert: one row with `req_id == "LLR-0001"`, `criterion_index == 2`.

### TC-002 — audit_criteria reports unlinked criteria
Seed an LLR with three `- [ ]` criteria items but only one `CRITERION:` tag (for `#1`).
Call `engine.audit_criteria()`. Assert: `CriteriaReport` shows `linked: true` for index 1,
`linked: false` for indices 2 and 3.

### TC-003 — audit_criteria reports all criteria as linked when fully covered
Seed an LLR with two criteria; provide `CRITERION:` tags for `#1` and `#2`.
Assert: both entries show `linked: true`.

### TC-004 — max #N exceeds criterion count produces Warning
Seed an LLR with 2 criteria. Provide a `CRITERION: LLR-XXXX #5` tag.
Call `audit_criteria()`. Assert: the report contains a warning about the out-of-range index.

### TC-005 — Invalid CRITERION tag format is ignored without panic
Write a source file with `// CRITERION: BADFORMAT x`.
Run `engine.scan(None, false)`. Assert: no panic, no entries in `criterion_links`.

### TC-006 — Schema migration from version 1 to version 2 completes without data loss
Open an existing v1 database (no `criterion_links` table), run `ReqEngine::open`.
Assert: `criterion_links` table now exists; existing `code_refs` rows are intact.

## Acceptance Criteria

- [ ] TC-001 through TC-006 pass in the default build
- [ ] Existing `REQ:` and `VERIFIES:` tag handling is unaffected (verified by running
      full existing test suite)
