## Description

Tests verifying that `req scan` stores file hashes and uses them to skip
unchanged files on subsequent scans while re-parsing changed files.

## Test Cases

### TC-0040-01 — Hash stored after first scan

**Given** a project with a source file containing a `// REQ:` tag
**When** `scan()` is executed for the first time
**Then** a non-empty SHA-256 hash for that file is present in the `file_hashes`
table in the cache

### TC-0040-02 — Unchanged file skipped on second scan

**Given** a project where a first `scan()` has already stored hashes
**When** `scan()` is executed again without modifying any source files
**Then** the result is identical to the first scan and the stored hashes are
unchanged

### TC-0040-03 — Modified file re-parsed

**Given** a project with stored file hashes from a previous scan
**When** a source file is modified (adding a new `// REQ:` tag) and `scan()` is
re-run
**Then** the new code reference appears in the cache and the stored hash is
updated to the new value

### TC-0040-04 — `clear=true` forces full re-scan

**Given** a project with stored file hashes
**When** `scan(None, true)` is executed (clear flag)
**Then** all previously stored hashes are removed before the scan, every file
is re-parsed, and the hashes are repopulated after the scan

### TC-0040-05 — Incremental result matches full re-scan result

**Given** a repository in a fixed state
**When** a full `scan(None, true)` is performed, followed by an incremental
`scan(None, false)` without any file changes
**Then** both scans report the same `refs_found` count
