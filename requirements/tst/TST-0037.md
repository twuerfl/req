## Description

Unit and integration tests verifying that lock contention on `cache.db` always produces
a non-zero exit code and a human-readable error message, and that `--wait` retries
correctly.

## Test Cases

### TC-0037-01 — `req init` on a locked database exits non-zero

**Given** `cache.db` is held open in exclusive write mode by a second process
**When** `req init` is executed (without `--wait`)
**Then** the command exits non-zero and stderr contains the string `cache.db is locked`

### TC-0037-02 — Error message names the likely holder

**Given** the same locked-DB setup as TC-0037-01
**When** any write command is executed
**Then** stderr contains a reference to the MCP server as the likely lock holder and
suggests using `--wait`

### TC-0037-03 — `req scan` on a locked database exits non-zero

**Given** `cache.db` is held open in exclusive write mode
**When** `req scan` is executed
**Then** the command exits non-zero and stderr contains the lock error message

### TC-0037-04 — `--wait 0` (default) fails immediately

**Given** a locked database
**When** `req scan --wait 0` is executed
**Then** the command fails immediately without sleeping and exits non-zero

### TC-0037-05 — `--wait <n>` retries and succeeds on lock release

**Given** a locked database that will be released after ~200 ms
**When** `req scan --wait 2` is executed
**Then** the command retries, acquires the lock after release, completes successfully,
and exits 0

### TC-0037-06 — `--wait <n>` times out and exits non-zero

**Given** a database that remains locked for longer than the wait timeout
**When** `req scan --wait 1` is executed
**Then** the command exits non-zero after approximately 1 second and stderr contains the
lock error message

### TC-0037-07 — No silent swallowing of SQLITE_BUSY in engine layer

**Given** a mock rusqlite connection that always returns `SQLITE_BUSY`
**When** any engine-layer write function is called
**Then** the function propagates an error — it does not return `Ok(())` or `Ok(0)`
